# 葡萄通：面向葡萄园病害预警与精准管理的端到端移动决策支持平台

## 摘要

葡萄生产面临病害高发、气象驱动风险变化快、田间管理时效性强和农事记录分散等问题。传统病害监测主要依赖人工巡园和经验判断，难以及时整合天气、物候、品种感病性、历史施药、图像识别和田块空间信息，从而限制了精准防控和绿色用药决策的实施。本文提出并实现了一套面向葡萄园管理的端到端数字化平台“葡萄通”。该平台集成移动端田块管理、天气数据接入、葡萄物候模拟、病害风险模型、施药保护期评估、农事任务管理、图像识别、通知预警和论坛交流等功能，形成从数据采集、模型计算、风险评估到用户决策反馈的完整闭环。

系统采用 Flutter 移动端、Django REST Framework 后端、PostgreSQL 数据库和独立 FastAPI 病害模型服务的分层架构。后端周期性获取地块尺度天气数据，结合品种、栽培方式、树龄和作季信息运行物候模型，并将逐小时天气、生育期、历史施药记录和病害目标输入病害风险模型，输出逐日感染风险、病害风险、田块综合风险、药剂保护期、处理建议和防治窗口。平台同时集成 YOLO、ViT 和分割模型相关接口，用于葡萄病害图像识别、病害上报和严重度辅助判断。通过移动端应用，用户可完成地块绘制、作季建档、天气查看、病害预警、植保与灌溉任务管理、农事记录和病害交流。

本文重点描述葡萄通的系统架构、数据流程、模型集成方式和应用功能，并讨论其作为葡萄病害预警与精准管理平台的技术特点、应用价值与当前局限。该系统表明，将气象驱动病害模型、移动端田间工作流和图像智能识别集成到统一平台，有助于提升葡萄园病害风险感知、管理建议生成和农事执行记录的数字化水平。

**关键词**：葡萄；病害预警；精准农业；移动决策支持；物候模型；病害风险模型；图像识别；Django；Flutter

## 1 引言

葡萄是重要的经济作物，其生产过程受到霜霉病、白粉病、灰霉病、黑腐病等多种病害威胁。病害发生与温度、湿度、降水、露点、作物生育期、品种感病性、栽培方式和施药历史密切相关。实际生产中，病害防控往往需要在短时间内完成风险判断和管理决策。如果仅依赖人工巡查和经验判断，容易出现预警滞后、施药时机不合理、药剂保护期估计不足或重复用药等问题。

近年来，精准农业和智慧农业技术为葡萄园管理提供了新的路径。天气服务、地理信息系统、移动终端、机器学习、图像识别和作物病害模型可以为生产者提供更及时、更细粒度的决策依据。然而，在实际应用中，许多系统仍停留在单一功能层面，例如单独的天气查询、病害图像识别、作物模型计算或农事记录工具。对于生产者而言，更关键的需求是将田块、作季、天气、物候、病害风险、施药任务和田间记录组织为同一套可持续运行的工作流。

类似 Bakir 等关于精密牧场平台的研究，农业平台型系统的价值不仅在于某一个算法或传感器，而在于能否建立从数据接入、模型分析、数据库管理、用户界面到现场部署的端到端架构。基于这一思路，本文以“葡萄通”为研究对象，系统整理其面向葡萄园病害预警与管理的总体架构和实现方式。

本文的主要目标包括：

- 构建面向葡萄园病害风险预警和田间管理的端到端平台架构。
- 集成天气数据、物候模型、病害模型、历史施药记录和品种感病性信息，形成作季级风险评估流程。
- 通过移动端应用支持田块管理、任务管理、病害识别、通知预警、农事记录和社区交流。
- 将传统病害模型和深度学习图像识别模型接入统一后端，为生产者提供多源信息驱动的辅助决策。
- 总结该平台当前的技术特点、局限性和后续研究方向。

## 2 系统总体架构

葡萄通采用多层解耦架构，整体可分为四个核心层：移动应用层、后端业务与数据管理层、模型分析层和外部数据接入层。

### 2.1 移动应用层

移动端基于 Flutter 实现，面向 Android 设备提供统一用户入口。应用主界面由四个核心模块组成：

- 地块：用于地块创建、边界绘制、作季管理、天气查看、通知查看和地块共享。
- 任务：用于植保任务、灌溉任务、任务状态和作业执行记录管理。
- 识别：用于病害图片上传、智能识别、病害上报和历史消息查看。
- 论坛：用于用户发帖、评论、回复和交流。

移动端通过 HTTP API 与 Django 后端通信，并通过 WebSocket 接收通知和智能识别会话消息。应用集成高德地图、定位、图片选择与压缩、PDF 查看、视频播放、本地存储和图表展示等功能，支持田间场景下的数据录入和结果查看。

### 2.2 后端业务与数据管理层

后端基于 Django 4.2 和 Django REST Framework 构建，使用 PostgreSQL 存储用户、田块、作季、天气、物候、病害、任务、通知、论坛、农事记录和基础资料等结构化数据。后端同时提供 REST API、Swagger/Redoc 文档接口和 Channels WebSocket 通信能力。

后端主要业务模块包括：

- 用户模块：手机号、验证码、用户信息和登录状态管理。
- 地块模块：地块 GeoJSON、面积、质心、边界图、共享和回收站。
- 作季模块：葡萄品种、栽培方式、树龄和地块关联。
- 基础数据模块：葡萄品种、杀菌剂、杀虫剂、生育期、病害类型、栽培方式和提醒规则。
- 天气模块：天气数据获取、喷药适宜性计算、作季天气数据保存和周期性更新。
- 物候模块：葡萄生育期模拟和物候数据保存。
- 病害模块：病害模型请求体保存、病害风险结果保存和风险查询。
- 任务模块：植保任务、灌溉任务、执行状态和历史施药记录。
- 通知模块：病害风险、生育期和任务相关提醒。
- 图像识别模块：YOLO、ViT、分割模型等识别服务接口。
- 社区模块：论坛、评论、回复和通知。

### 2.3 模型分析层

模型分析层包括三类核心模型：

1. 物候模型
   - 根据作季位置、年份、品种、树龄和逐日平均温度等输入，模拟葡萄生育期。
   - 输出 GDD、累计 GDD、BBCH 主阶段和一位数生育期阶段。
   - 结果用于病害模型中的生育期约束和生产管理提醒。

2. 病害风险模型
   - 独立部署为 FastAPI 服务，接口为 `POST /simulate`。
   - 接收作季级输入，包括天气、物候、生育期、历史施药、病害目标、品种感病性和请求的结果类型。
   - 输出逐日感染风险、短周期聚合风险、病害风险、田块综合风险、药剂保护期、处理建议和防治窗口。
   - 当前葡萄病害支持霜霉病、白粉病、灰霉病、炭疽/黑痘相关病害和黑腐病等。

3. 图像识别与分割模型
   - 后端集成 YOLO、ViT/EfficientViT、UNet 或分割模型相关代码和权重。
   - 支持病害图像识别、病害发现上报和严重度文本输出。
   - 图像模型结果可作为用户巡园和病害记录的补充信息。

### 2.4 外部数据接入层

平台接入多类外部服务：

- 天气历史数据服务。
- 未来天气预报服务。
- 经纬度反查行政区域服务。
- 短信验证码服务。
- 病害模型线上服务代理。
- WebSocket 通知和智能识别服务。

这些外部服务为平台提供天气、地理位置、认证和模型能力，但也引入了网络稳定性、服务可用性和配置管理方面的依赖。

## 3 数据模型与主流程

### 3.1 田块与作季数据

地块是平台的空间管理单元。用户在移动端创建地块并绘制边界后，系统保存 GeoJSON 边界，计算地块面积、质心经纬度，并生成 SVG 边界图。系统进一步通过地理编码接口获取地块所属区域。

作季是病害、天气、物候和任务数据的核心关联对象。每个作季关联一个地块、葡萄品种、栽培方式和树龄。葡萄品种包含熟期、用途、品种类型、抗性描述、霜霉病感病性和白粉病感病性等信息。这些参数会在病害风险模型中作为重要输入。

### 3.2 天气数据处理

后端周期性调度任务遍历所有未删除地块下的作季，并根据作季地块质心经纬度请求天气服务。天气数据包括历史天气和未来 16 天预报，主要变量包括：

- 逐小时温度。
- 相对湿度。
- 露点。
- 降水。
- 风速。
- 风阵。
- 逐日最高温、最低温、平均温。
- 逐日降水量。
- 日出和日落时间。

系统对缺失天气值进行简单填补，并进一步计算喷药适宜性。逐小时天气数据会被转换为病害模型所需字段，包括时间、露点、降水、相对湿度、气温和风速。

### 3.3 物候模拟

系统以逐日平均气温为主要输入，调用物候模型生成作季内每日生育期结果。输出包括：

- 日期。
- 当日 GDD。
- 累计 GDD。
- BBCH 主阶段。
- BBCH 一位数阶段。
- 生育期中文和英文描述。

物候结果一方面用于移动端展示，另一方面用于病害模型判断病害发生期和防控提醒。

### 3.4 病害风险模拟

病害模型请求体以作季为单位构造，关键字段包括：

- `crop_season_uuid`：作季标识。
- `province`：区域参数。
- `crop_eppo_code`：作物 EPPO 编码，葡萄为 `VITVI`。
- `time_zone`：时区。
- `latitude` 和 `longitude`：地块位置。
- `request_datetime`：请求日期。
- `growth_stage`：每日生育期数据。
- `weather_hourly`：逐小时天气数据。
- `applied_fungicides`：历史施药记录。
- `stress_eppo_codes`：目标病害。
- `variety_susceptibility`：品种感病性。
- `requested_features`：请求返回的结果类型和时间范围。

平台主要请求以下结果：

- `dailyInfectionRisks`
- `stressRisks`
- `fieldRisks`
- `stressProtectionTimes`
- `actionRecommendations`
- `treatmentWindows`

模型返回后，后端将不同病害和田块综合风险整理为面向移动端展示的数据结构，并写入病害结果表。对于已完成的植保任务，系统会根据药剂基础数据中的靶标病害、保护天数、治疗天数和预防效果，生成历史施药记录并重新模拟风险。

### 3.5 通知与任务闭环

系统根据物候阶段、病害风险和处理窗口生成通知。用户可以在移动端查看通知，进入对应地块或任务页面，并创建或更新植保、灌溉等农事任务。任务状态变化后，尤其是植保任务完成或删除时，会影响历史施药记录，从而进一步影响病害模型中的药剂保护期和后续风险评估。

这一流程形成了“天气数据更新-物候模拟-病害风险模拟-通知提醒-任务执行-施药记录更新-风险重算”的闭环。

## 4 平台功能

### 4.1 地块管理

用户可在地图上创建地块、绘制边界、查看地块列表、编辑地块信息、共享地块、恢复或清理删除地块。地块边界以 GeoJSON 保存，并可生成 SVG 缩略图用于列表展示。

### 4.2 作季与葡萄品种管理

用户可为地块创建作季，并选择葡萄品种、栽培方式和树龄。品种感病性和栽培方式作为模型输入参与病害风险和喷药适宜性计算。

### 4.3 天气与喷药适宜性

平台展示作季关联地块的逐日和逐小时天气信息，并计算未来时段喷药适宜性。喷药适宜性综合考虑温度、降水、风速、相对湿度、未来短时降水、栽培方式、日出和日落时间。

### 4.4 病害风险预警

病害风险模块以图表和列表形式展示霜霉病、白粉病等风险结果，包括逐日感染风险、病害风险、田块综合风险、防治窗口和处理建议。系统可根据风险状态生成通知，提醒用户关注高风险日期或错过的处理窗口。

### 4.5 植保与灌溉任务管理

任务模块支持植保任务和灌溉任务的创建、查询、更新、删除和状态管理。植保任务可以关联杀菌剂，任务完成后相关药剂信息会进入历史施药记录，影响病害模型的药剂保护期评估。

### 4.6 农事记录

平台支持普通农事记录、生育期记录和病害发生记录。记录可关联作季、地块、图片和文字说明，为后续病害诊断、模型验证和生产追溯提供数据基础。

### 4.7 图像识别与病害上报

图像识别模块允许用户上传葡萄相关图片，后端调用目标检测、分类或分割模型，返回识别结果、处理后图片和严重度信息。该模块可辅助用户发现病害症状，并与病害上报、历史消息和智能问答模块结合。

### 4.8 论坛与知识交流

论坛模块支持发帖、评论、回复和用户通知，有助于种植者之间交流病害、防控措施和田间管理经验。

## 5 系统实现

### 5.1 前端实现

移动端采用 Flutter 和 Dart 开发。项目结构按功能模块划分，主要包括 `auth`、`field`、`task`、`communition`、`forum`、`map` 和 `note`。网络通信使用 `dio` 和 `http`，本地信息保存使用 `shared_preferences`。地图和定位能力由高德地图 Flutter 插件提供，图像上传依赖图片选择和压缩组件。

### 5.2 后端实现

后端采用 Django REST Framework 暴露 API，使用 PostgreSQL 保存业务数据。系统通过 Django APScheduler 执行周期性天气更新、物候模拟、病害模拟和通知清理任务。WebSocket 通信通过 Django Channels 实现。

### 5.3 病害模型服务实现

病害模型服务独立于 Django 后端，使用 FastAPI 提供 `/simulate` 接口。服务内部通过 `CropSeason` 对象组织天气、生育期、作物、病害和管理措施，并根据 `requested_features` 调用不同模型方法。该设计使病害模型可以作为独立服务部署，也便于未来扩展到更多作物和病害。

### 5.4 模型和业务服务耦合方式

Django 后端在业务流程中组装病害模型请求体，并通过 HTTP 请求调用病害服务。模型结果被持久化到数据库，再由移动端通过业务 API 查询。与在用户请求时即时计算相比，周期性计算和持久化结果更适合移动端快速展示和通知生成，但会带来一定时间延迟。

正式论文中可将验证设计分为三类：

1. 系统可用性验证
   - 统计平台在真实葡萄园中的运行天数、地块数量、用户数量和数据记录量。
   - 评估主要 API 和移动端功能是否稳定运行。

2. 模型结果合理性验证
   - 将病害模型输出风险与田间病害记录进行对比。
   - 分析高风险日期与实际发病或防治任务之间的时间关系。

3. 用户工作流验证
   - 记录用户从地块建档、天气查看、风险接收、任务创建到任务完成的闭环行为。
   - 通过问卷或访谈评估系统对管理效率和决策信心的影响。

## 7 讨论

葡萄通的主要特点是将多类与葡萄园管理相关的数据和模型集成到同一平台，而不是提供单点工具。其端到端链路包括地块空间数据、天气数据、物候模拟、病害风险模型、施药记录、任务管理和移动端提醒。这种架构使病害模型结果能够直接进入用户工作流，并通过任务状态变化反过来影响后续模型输入。

与单独的病害模型或图像识别工具相比，葡萄通的优势在于：

- 模型输入与田块、作季、品种和施药记录绑定，更贴近实际管理单元。
- 病害风险结果不仅展示给用户，还进一步触发通知和任务管理。
- 移动端降低了田间数据录入和结果查看门槛。
- 图像识别、病害记录和论坛交流为模型风险结果提供补充信息。
- 系统采用独立病害模型服务，便于未来扩展作物、病害和部署方式。

同时，系统仍存在若干局限：

- 前端和后端存在较多硬编码公网地址，不利于多环境部署和维护。
- 后端天气更新、物候模拟、病害模拟和通知生成逻辑耦合较强，后续应拆分为更清晰的异步任务和服务边界。
- 当前 Channels 使用内存层，不适合多进程或分布式部署。
- 图像识别模型的训练数据、验证指标和田间泛化能力需要进一步整理。
- 病害模型的田间验证数据、风险准确性和管理建议有效性仍需系统评估。
- 系统对外部天气服务、地理编码服务和短信服务依赖较强，需要更完善的容错和降级机制。

## 8 结论

本文整理并描述了一套面向葡萄园病害预警与精准管理的端到端移动决策支持平台“葡萄通”。该平台将 Flutter 移动端、Django 后端、PostgreSQL 数据库、FastAPI 病害模型服务、物候模型、天气服务、图像识别模型和 WebSocket 通知集成到统一架构中，支持地块管理、作季建档、天气查看、物候模拟、病害风险预警、植保任务、灌溉任务、农事记录、图像识别和论坛交流。

葡萄通的核心价值在于打通了从田块空间数据和天气数据到病害模型输出，再到通知提醒和任务执行记录的管理闭环。该平台为葡萄病害智能预警、绿色防控和田间数字化管理提供了可扩展的软件基础。未来研究应重点补充真实部署数据、模型验证结果、用户行为分析和系统性能评估，并进一步提升配置管理、服务解耦、容错能力和多环境部署能力。
